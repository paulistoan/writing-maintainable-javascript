Mediator

“Define an object that encapsulates how a set of objects interact. Mediator promotes loose coupling by keeping objects from referring to each other explicitly, and it lets you vary their interaction independently.”
As the number of objects in a system increases, so do the interconnections between them. This works against reusability. The mediator pattern can help us with this by decoupling components and funnelling all their communication through a common mediator.

Conclusions

  • Program to interfaces, not to implementations.
  • Favor composition over inheritance.
  • Program to interfaces
    • Easy to swap out implementations, configurable
    • Maintainable
  • Favor composition
    • Inheritance is rigid
    • Composition allows finer grained control over what parts to implement in what modules
    • Composition allows us to swap out one functionality for another with very little cost
    • Composition forces us to think about the public interface of an object while inheritance allows us to tinker with its internals which breaks encapsulation.
